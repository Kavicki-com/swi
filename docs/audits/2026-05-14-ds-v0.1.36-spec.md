**Branch (consumer):** `feat/admin-maps-general-states`

## Why this bump

The hex-literals audit (`docs/audits/hex-literals-audit-2026-05-14.md`)
identified 12 hardcoded colour values in `swi-admin/src/pages/` that **do
not map to any existing token** in v0.1.35. After the 6 direct swaps
landed in this branch, the residual 12 occurrences must either:

1. Get a DS token added in v0.1.36 (this spec), then swapped consumer-side
   in a follow-up PR; or
2. Be documented inline as deliberate exceptions.

This spec covers option 1 for all 12 occurrences.

## Proposed tokens — grouped

### 1. Status palette (NEW namespace: `theme.status.*`)

Worker condition palette used on map pins and KPI tiles. **Intentionally
distinct from `theme.surface.{success,warning,error}`**: the existing
surface palette uses lime/yellow/red branded values (e.g. `surface.success
= lime[700] = #3EAB2E`) which are tuned for fills/CTAs. Worker-status pins
on a satellite basemap need the brighter web-standard greens/oranges/reds
that the Figma chose (Tailwind-ish hues) so they read instantly against
photographic terrain.

```ts
// semantic.ts
status: {
  good:    primitive.statusGreen[500],   // #10B981 — worker on-shift, vitals OK
  alert:   primitive.statusOrange[500],  // #F59E0B — vitals warning
  low:     primitive.statusRed[500],     // #EF4444 — critical, intervene
  offline: primitive.neutral[400],       // #7F7F7F — re-use existing neutral
},
```

New primitive ramps required (`primitive.ts`):

```ts
statusGreen:  { 100: '#D1FAE5', 300: '#6EE7B7', 500: '#10B981', 700: '#047857' },
statusOrange: { 100: '#FEF3C7', 300: '#FCD34D', 500: '#F59E0B', 700: '#B45309' },
statusRed:    { 100: '#FEE2E2', 300: '#FCA5A5', 500: '#EF4444', 700: '#B91C1C' },
```

Only the 500 step is consumed in v0.1.36; ramps shipped to give future
work (status badges, severity backgrounds) room to grow without another
primitive bump. `offline` reuses `neutral[400] = #7F7F7F` instead of the
literal `#6b7280` — neutral[400] is one shade lighter but reads identically
on a dark basemap and avoids a 5-shade neutral grey just for this one pin.

### 2. Surface — route states (extend `theme.surface.*`)

The rescue-route screen shifts colour after the operator hits "Continuar":
cyan (pre-dispatch) → violet (dispatched, walking marker active). Both
strokes need to read on a satellite basemap so they don't share with
`surface.secondary` (info blue) or `surface.accent` (yellow).

```ts
// semantic.ts → surface
routeActive:    primitive.cyan[400],    // #2BA8C9 — rescue route, en route
routeDispatched: primitive.violet[500], // #8B5CF6 — rescue route + walker marker, dispatched
```

New primitive ramps:

```ts
cyan:   { 100: '#CFFAFE', 300: '#67E8F9', 400: '#2BA8C9', 600: '#0891B2' },
violet: { 100: '#EDE9FE', 300: '#C4B5FD', 500: '#8B5CF6', 700: '#6D28D9' },
```

### 3. Gradients (NEW namespace: `theme.gradients.*`)

Charts read smoothly between two stops. Existing tokens like
`surface.success` are solid; charts need explicit pairs.

```ts
// semantic.ts
gradients: {
  wear: [primitive.statusGreen[300], primitive.statusGreen[500]] as const,
    // #6EE7B7 → #10B981  (fatigue chart — light to deep)
  urgent: [primitive.statusRed[300], primitive.urgentDeep[500]] as const,
    // #FCA5A5 → #F5667A  (urgent alerts chart — light pink to salmon)
},
```

The `urgent` "deep" stop is the Figma-specified `#f5667a` (salmon-coral),
which isn't on the statusRed ramp. Add as a one-shot primitive:

```ts
urgentDeep: { 500: '#F5667A' }
```

Or, simpler: keep `urgent[1]` as a literal in the gradient definition and
skip the primitive entry — `urgentDeep` is a chart-specific colour, not a
reusable shade. The DS author should pick.

### 4. Content — alert overlay (extend `theme.content.*`)

Light-pink label drawn over the meteorologic basemap. Needs to be
distinctive against red-tinted basemaps where `content.dark` (#F5F5F5)
gets lost.

```ts
// semantic.ts → content
alertOverlay: primitive.statusRed[300],   // #FCA5A5 — over-image alert label
```

Audit recorded `#fda4af`; statusRed[300] is `#FCA5A5` — 1-LCH-step
difference, visually indistinguishable. Reusing the existing primitive
avoids adding `pink: { 300: '#FDA4AF' }` for one consumer.

### 5. Weather intensity (NEW namespace: `theme.weather.*`)

Weather timeline rain segments use a saturated denim cyan distinct from
both `routeActive` (cyan, but pure web cyan) and `surface.info` (info
blue). Make this a small dedicated palette so a future "sunny", "cloudy",
"storm" segment can join it without polluting `surface.*`.

```ts
// semantic.ts
weather: {
  rain:        primitive.weatherBlue[500],  // #3899BF
  sunny:       semantic.surface.warning,    // existing — used today for seg-1
  partlyCloudy: semantic.surface.success,   // existing — used today for seg-3
},
```

New primitive:

```ts
weatherBlue: { 300: '#7CC1D9', 500: '#3899BF', 700: '#1C6E8F' }
```

`sunny` and `partlyCloudy` re-use existing tokens that already match the
Figma weather segments (`#EF8600` and `#3EAB2E`, both already swapped this
branch). So `theme.weather` is mostly a semantic alias layer.

## Open design questions

1. **`offline` exact value.** Audit hex is `#6b7280` (Tailwind slate-500);
   spec proposes `neutral[400]` (`#7F7F7F`). Need to confirm one-shade
   lighter is acceptable, or add a new `primitive.coolGrey: { 500: '#6B7280' }`.

2. **`urgentDeep` primitive vs literal.** Add a 1-key primitive
   (`urgentDeep: { 500: '#F5667A' }`) or keep the hex inline in the
   gradient definition? Asking because the urgent chart is the only
   current consumer.

3. **`status` vs `pin`.** Considered naming the namespace `theme.pin.*`
   but status is broader — these colours will appear in non-pin contexts
   (severity dots, KPI badges, status chips). `status` reads cleaner.

4. **`weather.sunny` / `weather.partlyCloudy` as aliases.** The proposed
   semantic block aliases `surface.warning` and `surface.success` under
   weather names so weather code reads `theme.weather.partlyCloudy`
   instead of `theme.surface.success`. Pro: domain clarity. Con: aliases
   can drift if the weather designer decides on a different green later.
   DS author should pick.

## Implementation steps (DS repo)

1. Edit `swi-design-system/src/tokens/primitive.ts` — add `statusGreen`,
   `statusOrange`, `statusRed`, `cyan`, `violet`, `weatherBlue` ramps.
   Optionally `urgentDeep`. **Do NOT touch the generated `figma-variables.json`**
   — these new tokens originate in code, not Figma, and the generator runs
   one-way. (If Figma later adds matching variables, the generated values
   take over.)
2. Edit `swi-design-system/src/tokens/semantic.ts` — extend `surface`
   with `routeActive` + `routeDispatched`; add `status`, `gradients`,
   `weather`, `content.alertOverlay`. Keep alpha sorting consistent with
   the file's existing groups.
3. `npm run build` — verify dist + types still emit clean.
4. `npm run lint` — block any unused-token warnings if introduced.
5. Storybook: add `tokens/_docs/Status.stories.tsx`,
   `tokens/_docs/Gradients.stories.tsx`, `tokens/_docs/Weather.stories.tsx`
   (or extend the existing Semantic story with the new groups). Document
   the visual shape so design can confirm before publish.
6. Bump `package.json` to `0.1.36`. `npm publish` (or local pack).
7. swi-admin: `npm install @kavicki/swi-design-system@0.1.36` and land
   the migration PR from the table above.

## Risk

- **Low** structural risk: only token additions, no renames or breaking
  changes to existing exports.
- **Medium** design risk: the proposed primitive ramps (statusGreen,
  statusOrange, statusRed) are Tailwind-derived. If the SWI brand has
  formal colour specs for severity states, those should win over these
  proposals.
- **Zero** consumer-side risk for swi-admin during the bump itself —
  swaps happen in a follow-up PR after the bump lands.
