**Date:** 2026-05-08

## Real DS gaps that need a v0.1.4 bump

After re-auditing, only **one** real gap remains:

### Gap M — `MenuItem` styling does not match Figma sidebar

**Figma (frame `4:2` sidebar):**
- Compact rows ~40-48px tall
- **Uppercase** labels ("HOME", "MAPAS", "ALERTAS")
- Active item: **green text + 2px green left bar**, **no background fill**
- Inactive item: light grey text, no bar

**DS v0.1.3 `MenuItem`:**
- Container `height: 64px` (too tall)
- Active state has `surface.standard` **background fill** (filled grey pill) — Figma has none
- `Divider` is rendered on the **right** side of the row (Figma has it on the left)
- Label is `font-weight: bold, font-size: m` — no uppercase / no letter-spacing tuning
- `flex-direction: row; justify-content: space-between` puts label on left and the bar on the right

**Decision needed.** Three paths:

| Option                                         | Cost | Pros                                                                 | Cons                                                                 |
|------------------------------------------------|------|-----------------------------------------------------------------------|----------------------------------------------------------------------|
| **A. Add `variant: 'default' \| 'compact'` prop** to `MenuItem` (and pass-through on `SideMenuItem`) | Low — one prop + a styled-component branch | Clean, reusable, contained in DS | Needs a v0.1.4 release + bump in `swi-admin` |
| B. Override `MenuItem` styles via wrapper in `swi-admin` | Lowest — 1 file in admin | No DS release needed; ships today | Local CSS escape hatch; future styles drift; design system loses authority |
| C. Replace `SideMenu` in `swi-admin` with a custom local component | Low | Full control | Re-implements `accessibilityRole="menu"` + active state + hover; bigger blast radius; defeats DS |

**Recommendation: Option A.** The DS bump is genuinely small (20-40 lines of styles + one prop) and resolves the only remaining DS gap. After this, the dashboard rework is pure `swi-admin` work.

#### Proposed `MenuItem` v0.1.4 API

```ts
// MenuItem.types.ts
export type MenuItemVariant = 'default' | 'compact';

export interface MenuItemProps {
  label: string;
  icon?: IconName;
  active?: boolean;
  disabled?: boolean;
  variant?: MenuItemVariant; // NEW — default 'default' (no breaking change)
  onPress?: () => void;
  fullWidth?: boolean;
  accessibilityLabel?: string;
  testID?: string;
}
```

Pass-through on `SideMenuItem` and `SideMenuProps.variant`.

#### Compact variant style targets (from Figma reading)

| Token                   | `default`                | `compact`                                                |
|-------------------------|--------------------------|----------------------------------------------------------|
| `Container.height`      | 64px                     | **44px**                                                 |
| `Container.background`  | `surface.standard` on active / `background` otherwise | **transparent on all states** |
| `Container.padding`     | `padding.m` left, `padding.s` right | **`padding.s` both sides** |
| `Container.flex-direction` | row (label left, divider right) | **row** but **divider rendered as the FIRST child** (left side) |
| `Label.font-size`       | `fontSize.m` (~14px)     | **`fontSize.s` (~12px)**                                  |
| `Label.text-transform`  | none                     | **uppercase**                                             |
| `Label.letter-spacing`  | normal                   | **0.6-0.8px**                                             |
| `Divider.position`      | right of row             | **left of row, height 100%, width 2px**                   |
| `Divider.color`         | accentColor              | accentColor (unchanged — already green when active)       |
| `HoverOverlay`          | full bg                  | unchanged (still subtle bg over the row)                  |

This is a contained styled-components branch; ~30 lines of code.

## Things that are NOT DS gaps — pure `swi-admin` rework

These are bugs/omissions in my S1.7 implementation that were misattributed to DS. To be fixed in Lote D:

1. **Logo position** — `AppLayout` uses DS `Header` which bundles Logo. Figma puts Logo in the sidebar top, not the header. Fix: render `<HeaderUserInfo>` directly in `AppLayout` (skip the `Header` wrapper) and put `<Logo>` at the top of the sidebar `View`.
2. **Map preview banner** — was a flat dark box. Fix: save the Figma map screenshot as `swi-admin/src/assets/dashboard-map-preview.png` and use DS `Image` + Button overlay. (Confirmed in user decision.)
3. **KPI row composition** — used 4 plain `BigNumbersCard`s. Fix: replace Sinais vitais + Taxa de desgaste with `DonutChart`. Compose Funcionários with two `BigNumbersCard`s + `StatusChart` + `AvatarGroup` inside one card surface.
4. **Sidebar missing chat list** — fix: render `<ChatSection users={...} />` inside the sidebar `View` after the `<SideMenu compact>`. Mock `ChatSectionUser[]` (4 entries match the Figma) — defer real chat to S5.
5. **Activities avatars rendering as red circles** — `pravatar.cc` URLs were probably blocked by CSP, dev network, or are simply broken. Fix: try `https://i.pravatar.cc/300?img=N`-style URLs or use local placeholder PNGs. Verify with browser load test.
6. **Activities leading icon** — `ActivitiesOverviewCard` already defaults to `icon='build'` (wrench). I just need to drop my override or pass an explicit per-activity icon.
7. **WeatherTimeline cramped** — likely a parent-container width issue. Fix: ensure the WeatherTimeline is in a `View style={{ alignSelf: 'stretch' }}` row that takes full width, and pass `intensitySegments` explicitly so the gradient bar renders deterministically.
8. **`Header` vs `HeaderUserInfo`** — the user-vitals widget in the top-right uses `HeaderUserInfo` directly (no Logo). DS `Header` wraps Logo + HeaderUserInfo together; that's wrong for this layout. Use `HeaderUserInfo` alone in `AppLayout`.
